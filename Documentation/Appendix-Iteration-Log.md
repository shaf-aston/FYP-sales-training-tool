## C.3 Measurement Methodology

**Stage accuracy** was calculated as (correct FSM transitions / total observed transitions) across 25 developer-executed test scenarios. Each scenario was run once with the same Groq model configuration and YAML product settings. "Correct" was defined as: (a) stage advanced only when advancement rules in `_check_advancement_condition()` were satisfied, and (b) no stage was skipped without meeting prerequisite signals.

**Question-per-turn counts** were measured by manual inspection of bot turn content in the same 25-scenario set, before and after each fix was applied. The baseline figure (avg. 3 questions/response) was captured pre-fix; achieved figure (max 1–2) reflects post-fix responses under P3 enforcement.

**Tone-match rate** (62% → 95%) was assessed across 12 developer personas (covering formal/casual registers, terse/verbose styles, and domain-specific vocabularies). A response was scored as tone-matched if it embedded at least one user keyword naturally and avoided register mismatch (e.g., formal reply to a casual opener).

**Negation false-positive rate** was assessed against 8 targeted test messages containing negated intent/doubt phrases. Zero false positives post-fix.

All baseline metrics were captured before the fix was applied; achieved metrics used the same scenario set with identical configuration to ensure comparability. The developer served as assessor throughout — evaluation of LLM output quality is inherently subjective at this scale and should be interpreted as directional evidence rather than statistically significant measurement.
