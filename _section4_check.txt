LLM Output VarianceLanguage models are probabilistic. Identical prompts yield different responses across runs. This is not a bug–it is the architecture. Consequence: The chatbot cannot guarantee identical behaviour in every situation, but staged constraints (YAML advancement rules, override checks) reduce variance to acceptable levels for training use.Empirical evidence: Across 170 domain tests with fixed prompts, completion rate variance ≈ ±2% turn–to–turn. Advancement stage remains consistent 94% of the time due to deterministic FSM rules.2. Context Window LimitationsThe LLM operates on a finite context window (~4k tokens per conversation). Extended conversations compress older context, potentially losing earlier problem statements. Mitigation: Session summaries are injected before each API call to maintain problem continuity.Addressable Limitations (Deliberate Scope Choices)LimitationImpactWhy Not FixedFor ProductionSession persistenceSessions stored in memory; lost on server restartTrade–off: File–based persistence adds latency (~50ms per write); in–memory is acceptable for training toolRequires database (PostgreSQL) or RedisInput validationPattern–matching filters only; regex does not catch all injection attemptsSufficient for this scope (non–adversarial, controlled trainer input); not suitable for public APIsRequires token–based input sanitisation + WAFSingle–server deploymentNo horizontal scaling; max ~50 concurrent sessions before degradationArchitectural choice: Flask app is stateless; sessions in–memory breaks distributed designRequires microservices + session storeProvider failure handlingNo retry logic; Groq timeout → immediate API error returned to userGroq uptime 99.8% (audited); retries add 2–3s per failure; not justified for training contextImplement exponential backoff + fallback provider queueIntent classificationAmbiguous phrasings (e.g., "maybe later" vs. "I need to think") remain misclassifiedWould require LLM–based classifier (adds 300ms latency per turn); keyword heuristic is 85% accurate for clear intentAcceptable for training; retrain classifier for productionKnown Gaps From Testing CoverageEdge cases in objection reframing (6/170 tests involved manual expectation–setting, not automated assertions)Quiz scoring fallback (LLM failure → score 50); no keyword–based fallback implementedProspect mode scoring (deterministic keyword–based scoring; no A/B ground truth from real salespeople)Inherent LLM constraints (variance, context limits) are well–understood and mitigated where possible. Hartmann et al. (2023) confirm that the intersection of sales and ethics is "vast and, therefore, difficult to summarize" spanning over four decades of unresolved scholarly debate – precisely because the same techniques that build trust can equally exploit it.The individual salesperson's moral judgment sits at the centre of this debate. Ingram et al. (2007) argue that “enhancing the moral behaviour of salespeople is increasingly important, but difficult " given that sales management decisions directly shape frontline ethical conduct.The system is deployed as a supplementary practice tool, designed to reward consultative behaviours – specifically active listening, open–ended questioning and empathy, rather than high–pressure or directive interaction patterns. Stages cannot advance without customer–signal detection, structurally incentivising discovery over persuasion.4.7 Future workVoice mode would benefit most from paralinguistic analysis: real–time detection of filler words, talk–to–listen ratio and pacing against stage norms would surface coaching signals unavailable in text–only sessions, extending the framework’s feedback depth into delivery as well as content.Performance persistence is a further priority: persisting analytics per user profile across sessions would enable the system to surface each practitioner’s highest–leverage improvement areas and track longitudinal progress, converting one–off practice into a structured development pathway.A sales call analysis feature, enabling users to upload or record real calls for automated framework–aligned feedback, would extend the system’s value beyond simulated practice into live performance review.A gamification layer would introduce XP awarded per session scaled by performance score, unlocking difficulty tiers and prospect variants as practitioners improve, with a team leaderboard surfacing rankings by weekly XP and stage progression to introduce the social accountability and motivation of an in–person sales floor. Crucially, all required inputs already exist in the deterministic scoring pipeline (score_session()), meaning implementation requires only a persistence layer and lightweight UI–making this among the highest–ROI extensions for a future iteration.4.8 Personal ReflectionI front–loaded too much time on designing the initial architecture before any implementation had been tested. Given that I naturally work through trial and error and manual testing, this was a poor fit for how I actually think. A tighter design–implement cycle from the start would have moved the project forward faster and grounded early decisions in real evidence rather than untested assumptions. A tighter build–test–redesign loop would have served the project better from the outset.If I could revisit any technical decision, it would be the move from keyword–matching to a cloud API provider. It would have been interesting to push further with keyword–matching (as this would be far more deterministic), augmenting it with NLP techniques for tighter framework adherence and introduced the LLM later purely as a corrective layer to pull conversations back on–track when they drifted or edge–cases. And in prospect–mode, LLM used only to create variance and unique feel. Personally, I have matured significantly. Investing heavily in prompt engineering, I developed a durable skill in controlled, structured and academically–backed prompt engineering while rooting myself in applying behavioural psychology.
